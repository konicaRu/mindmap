# project-starter — общий старт для новых проектов

**Запуск:** в новом проекте скажи — «**Начнём с project-starter**» (или «развернём по
project-starter»). Я читаю файл, провожу стартовый опрос (раздел 2) и разворачиваю проект.

**Две копии — держать синхронными:** `~/.claude/project-starter.md` (главная) и
`<корень проекта>/project-starter.md` (бэкап). Любую правку правил вношу сразу в ОБЕ копии.

> Переносимый шаблон правил, общих для всех моих проектов. Когда пользователь говорит
> «начнём с этого файла» (или «развернём по project-starter») — прочитай его целиком,
> проведи **стартовый опрос** (раздел 2), затем разверни проект: заведи контекстные файлы
> (раздел 7), при необходимости подключи опциональные модули (8, 9) и дальше работай по
> принципам ниже. Файл агент-facing: это инструкция для меня, агента.

## 1. Как использовать
1. Прочитать этот файл.
2. Задать вопросы из стартового опроса, дождаться ответов (не угадывать).
3. По ответам: создать `MEMORY.md`, `ARCHITECTURE.md`, `README.md`; при необходимости
   `git init`; подключить выбранные модули.
4. Записать в `MEMORY.md`/`README.md` цель и критерий готовности из ответов.
5. Дальше вести проект по принципам 3–7 + подключённым модулям.

## 2. Стартовый опрос (спросить пользователя, кратко)
- **Цель проекта** и **критерий готовности** (когда считаем задачу выполненной)?
- **Язык** общения и документов?
- **Git**: вести ли историю? есть ли remote (GitHub и т.п.) — пушим или только локально?
- **Модуль «накопительные правила»** (раздел 8) — нужен? (для проектов с итеративными
  правилами/замечаниями).
- **Humanizer** (раздел 9) — нужен? (для проектов, где на выходе текст). По умолчанию нет.
Предлагай модули сам, чтобы пользователь не забыл; решает — он.

## 3. Принципы работы (по Карпатому)
1. **Думай до действия.** При неоднозначности назови варианты и спроси, не выбирай молча.
   Замеченные противоречия выводи, а не сглаживай.
2. **Минимализм.** Делай ровно то, что решает задачу, без запасной «гибкости» и лишнего.
3. **Точечные изменения.** Меняй только нужное; каждое изменение прослеживается к команде.
4. **Работа от критерия.** У задачи есть критерий готовности — гоняй цикл, пока не выполнен.

## 4. Точечные правки — без каскада
Правка по конкретному месту — это точечное изменение, оно не должно «ломать всё»:
- Правь **хирургически** (`Edit` нужного фрагмента), не перезаписывай файл целиком ради
  одной правки, не трогай соседние/несвязанные части.
- Чини указанное + такой же огрех в **текущем** артефакте; уже готовое/отданное задним
  числом не переписывай.
- Не затирай ручные правки пользователя.
- **Не обобщай молча**: после правки спроси — разовая правка или общее правило? Правило
  фиксируй только по подтверждению («запиши / сделай правилом»).

## 5. Стиль работы / экономия токенов
- Отвечай кратко и по делу; без воды и пересказа того, что уже ясно.
- Внутренние рассуждения показывай **процентом готовности** («готовность 60%»), а не
  длинным нарративом.
- Не перечитывай файлы без нужды; не повторяй уже сделанную работу.
- Где есть индекс кода (codegraph/аналог) — используй его вместо grep для структурных
  вопросов; grep — для литерального текста.
- Независимые операции — параллельно (несколько вызовов в одном шаге).

## 6. Git-воркфлоу
- **`git save`** (в конце блока работы): (1) обновить контекстные файлы — дописать запись в
  `MEMORY.md` (сверху) и changelog в `ARCHITECTURE.md`, обновить «Статус»/«Открытые вопросы»;
  прошлое не переписывать, только добавлять; (2) `git add -A`, коммит; если есть remote —
  `git push`. Перед коммитом — sanity-check, что не утекли секреты/приватные данные
  (исключать через `.gitignore`).
- **`git load`** (в начале сессии): прочитать `MEMORY.md` + ключевые файлы и кратко доложить,
  где остановились. Только чтение.
- **История = откат.** Каждый `git save` — точка отката. Безопасно — `git revert`; один файл —
  `git checkout <commit> -- путь`; `git reset --hard` — только по явной команде пользователя.
- **Многострочный коммит** или сообщение со спецсимволами (`>`, кавычки, скобки) на Windows
  PowerShell писать через файл: `git commit -F .commitmsg.tmp`, затем удалить файл. Иначе
  here-string ломается и слова уходят в pathspec.

## 7. Контекстные файлы (заводить при старте, вести постоянно)
- **`MEMORY.md`** — журнал: «Статус проекта», «Открытые вопросы», «Лог сессий» (свежее сверху).
  Где остановились и почему. Обновляется по `git save`.
- **`ARCHITECTURE.md`** — карта проекта (структура, ключевые файлы, команды) + changelog
  (что и когда меняли). Правила работы не дублировать — только устройство и история.
- **`README.md`** — как пользоваться проектом: назначение, быстрый старт, команды; чтобы
  другой человек/агент мог войти.
Мини-шаблоны:
```
# MEMORY — журнал
## Статус проекта
## Открытые вопросы
## Лог сессий
### YYYY-MM-DD (тема)
```
```
# ARCHITECTURE
## Назначение
## Структура
## Команды
## Changelog
```

## 8. Опциональный модуль: накопительные правила
Для проектов, где правила копятся из замечаний (промпты, стайлгайды, политики):
- Замечание/правка → одно **атомарное правило** + скоуп (`always` или условие).
- Перед добавлением — **проверка на противоречия** с базовыми («ядро») правилами и активными.
  Конфликт не разрешать молча — показать и спросить.
- Журнал правил **append-only**: ничего не удалять; отменённое помечать статусом
  (`superseded`/`merged-into-core`); применять только активные.
- Базовый слой («ядро») менять только по явной команде; перенос правила в ядро — отдельным
  действием со сменой статуса.

## 9. Опциональный модуль: humanizer
Для проектов с текстом на выход (письма, посты, доки):
- Финальная шлифовка стиля: живой ритм, человеческий голос, без ИИ-штампов (пафос,
  негативные параллелизмы «не только…, но и», тройки ради ритма, синонимическая жвачка,
  канцелярит, длинные тире, дежурные концовки, филлеры).
- **Не меняет факты, структуру и обязательные требования** — только стиль.
- Состояние держать флагом (напр. строка `on`/`off` в файле). Включать по ответу в опросе.
- Если в среде есть скилл `humanizer` — использовать его; иначе применять принципы текстом.
